# 将 gemini-draw 迁移到其他网页登录型生成平台

`gemini-draw` 的核心价值，不只是 Gemini 本身，而是一套可复用的浏览器生成模式。

## 可复用模式

### 登录阶段

- 使用普通系统 Chrome
- 为每个平台使用独立 profile
- 不要使用自动化拉起的可见登录窗口

### 正式执行阶段

- 克隆该平台的模板 profile
- 启动私有 headless Chrome runtime
- 通过私有调试端口连接
- 不要碰用户正在使用的浏览器

### 验证阶段

- 从页面状态判断是否生成成功
- 触发原生下载或导出
- 验证文件是否真的落盘
- 验证文件大小或媒体元数据

## 每个平台会变化的部分

### 文生图平台

通常需要替换：

- 目标 URL
- 提示词输入框 selector
- 发送按钮 selector
- 图片完成判定
- 下载按钮 selector
- 文件类型预期

适用平台：

- 即梦
- 可灵图片
- 海螺图片
- 其他同类图像平台

### 文生视频平台

通常需要替换：

- 目标 URL
- 提示词输入框 selector
- 渲染完成判定
- 导出或下载触发方式
- 文件类型预期
- 视频时长或文件大小验证

适用平台：

- 海螺视频
- 可灵视频
- 即梦视频（如果支持）

## 平台特定风险

### 即梦

- 页面结构可能变化较快
- 需要显式检查登录态和额度状态
- 下载按钮可能藏在二级菜单里

### 可灵

- 图片和视频流程差异可能很大
- 生成过程通常需要更长轮询
- 预览出现后，导出按钮不一定立刻可用

### 海螺

- 视频完成常常晚于预览可见
- 导出可能会多一层确认按钮
- 文件审计最好同时检查容器格式和时长

## 泛化策略

如果以后要把这套模式迁移到其他网页登录型生成平台，建议保留一套共享 runtime，把平台差异收敛到各自的 provider 配置或说明里：

- URL
- selectors
- ready signals
- download signals
- audit rules

这样可以保持一套稳定 runtime，同时把平台差异隔离在薄薄的一层配置中。

## 推荐的未来形态

1. 保留 `gemini-draw` 作为稳定的 Gemini 专用实现
2. 如果扩展到其他平台，再在同一套浏览器生成架构上增加 provider 适配层

这样做的好处是：

- Gemini 主链保持稳定
- 其他平台可以沿用已验证过的登录、运行、下载和审计模式
- 不需要为每个平台从零再造一整套浏览器自动化架构
